# Skippy configuration setting matrix

This matrix turns the `report.md:144-339` inventory into the checked-in
operator reference for future schema and runtime wiring work. `report.md`
remains the authoritative inventory source, while this page defines the stable
config key paths, ownership expectations, mode support, validation intent, and
explicit rejection boundaries that later tasks must preserve.

The config file is a future-start and owner-control reload artifact, not a live
session mutation layer. Request defaults are merged only at the OpenAI frontend
boundary when request fields are absent or null, explicit request values win,
and the merged defaults never enter `StageConfig`, runtime load structs,
protobuf, or lower runtime layers.

## How to read this matrix

- `Config key path` is the canonical path later schema and wiring work must use.
- `Supported modes` only uses `single-stage`, `staged`, `request-only`, and
  `documented-rejected`.
- `Live-apply behavior` is either `request-time` or `restart/reload only`.
- `documented-rejected` means the key is intentionally reserved or the report row
  is intentionally excluded; operators should never expect a silent no-op.
- `Test evidence` points at the Task 1 checker outputs under
  `.sisyphus/evidence/`.

## Operator semantics

- Precedence is request values, then per-model config, then `[defaults.*]`,
  then family or topology policy, then built-in runtime defaults.
- Staged-only controls are gated at translation boundaries. Activation wire
  dtype, prefill controls, speculative draft controls, and manual stage layer
  ranges only execute in staged mode.
- Unsupported or deferred rows are intentionally marked `documented-rejected` or excluded from the matrix. Grammar, json_schema, logprobs, backend sampling bags, unsafe template override bags, and operational or security exclusions all live in that `documented-rejected` bucket.

## Deterministic macro mappings

- `model_fit.kv_cache_policy=quality` prefers `cache_type_k=f16`,
  `cache_type_v=f16`, `kv_offload=auto`, and no forced cache RAM cap.
- `model_fit.kv_cache_policy=balanced` preserves runtime and family defaults.
- `model_fit.kv_cache_policy=saver` prefers lower-memory cache dtypes, enables
  offload where supported, and must warn if the backend cannot honor that plan.
- `model_fit.kv_cache_policy=auto` lets family or topology policy choose; any
  explicit `cache_type_k` or `cache_type_v` value wins.
- `throughput.tuning_profile=throughput` biases toward larger batch or ubatch,
  enables continuous batching when safe, and raises `parallel` if the estimator
  says memory permits it.
- `throughput.tuning_profile=balanced` preserves today’s runtime defaults.
- `throughput.tuning_profile=saver` biases toward smaller batch or ubatch,
  lower `parallel`, and memory-saving KV defaults unless explicit fields win.
- `safety_margin_gb` is not a saved key here. Later wiring should derive
  `hardware.fit_target_mib` by subtracting reserved MiB from detected allocatable
  memory and must never write the resolved MiB value back into TOML.
