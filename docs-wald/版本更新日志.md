# mesh-llm 版本更新日志

本文记录本次三机启动 `Hermes-3-Llama-3.1-405B-Q8_0` 过程中，为 split GGUF 和 RPC tunnel 做过的关键版本更改。

## 0.65.0-rc2+splitgguf.1

### 版本目标

- 让 Mac host + 两台 PGX worker 能用本地 split GGUF shards 启动 405B Q8 模型。
- 避免 worker 明明有完整权重，却回退成通过 RPC tunnel 传输大 tensor。

### 碰到的问题

- 多机加载失败：`Remote RPC server crashed or returned malformed response`。
- 直接错误：`send failed (bytes_sent=0, size_to_send=285212976)`，随后 `SIGABRT`。
- 表面像 QUIC/RTT/大 payload 超时，实际是 split GGUF 本地 tensor 查找失败。

### 根因

- llama.cpp RPC 的 `SET_TENSOR_GGUF` 快路径只索引 `--gguf` 传入的第一个 shard。
- 后续 shard 中的 tensor 被误判为本地不存在。
- 本地查找失败后回退到原始 `SET_TENSOR`，把几百 MiB 到近 1 GiB 的 tensor 通过网络发送。

### 解决方案

- 新增 llama.cpp patch：索引同组全部 split GGUF shards。
- 读取 `split.count`，根据首个 shard 文件名推导并索引后续 shard。
- tensor 命中后由 worker 从本地 GGUF 读取，不再走大 tensor 网络传输。
- 临时放宽 RPC tunnel payload / timeout 上限，用于容错和排查。

### 验证记录

- 三台机器版本一致：`mesh-llm 0.65.0-rc2+splitgguf.1`。
- `/v1/models` 返回 `Hermes-3-Llama-3.1-405B-Q8_0`。
- chat completion 测试返回 `OK`。
- PGX RPC 日志确认从本地 shard 加载 tensor。

### 注意事项

- 这是 split GGUF + llama.cpp RPC 本地加载快路径的通用修复，不是 Hermes 专用。
- 每个 worker 仍需要有完整 GGUF shards，否则仍可能回退到网络传输。

## 0.65.0-rc2+splitgguf.5

### 版本目标

- 让 Mac host + 两台 PGX worker 形成稳定的三节点 RPC 互联。
- 修复 worker 之间缺少 B2B tunnel 时，`REGISTER_PEER` 指向 host 本地 `127.0.0.1:<port>` 的问题。
- 区分前端 `network error`、`3131` 管理接口、`9337` 模型接口和模型加载状态。

### 当前设计

- Mac 作为 host，启动 `llama-server`，监听 `3131` 管理接口和 `9337` OpenAI API。
- 两台 PGX 作为 split worker，各自启动本地 `rpc-server`，并从本机 GGUF shards 读取权重。
- host 传给 `llama-server` 的 `--rpc` 仍是 host 本地 tunnel port。
- worker 收到 `REGISTER_PEER` 后，将 host tunnel port 改写成本机可访问端口。
- 如果目标 peer 是 worker 自己，则改写成本机 `rpc-server` 端口。
- 如果目标 peer 是另一台 worker，则改写成本机到该 worker 的 tunnel port。

### 碰到的问题

- PGX 只连到 Mac，没有连到另一台 PGX。
- worker 收到 host 的 `127.0.0.1:<host_tunnel_port>` 后无法访问。
- rewrite map 在 tunnel map 到达前为空，首次 RPC 请求可能漏改写。
- 前端显示 `network error`，但 3131 实际正常，模型仍在加载。

### 解决方案

- `network/tunnel.rs`：支持把远端 tunnel map 中指向自己的端口改写成本地 `rpc-server` 端口。
- `network/tunnel.rs`：每次接收 inbound RPC tunnel stream 前刷新 rewrite map。
- `network/tunnel.rs`：tunnel watcher 增加首次扫描和 1 秒周期 reconcile。
- `inference/election.rs`：split 启动前主动 `regossip()`，让 worker 更快互相发现。
- `lib.rs`：版本号更新到 `0.65.0-rc2+splitgguf.5`，方便三台机器确认运行版本。

### 验证记录

- 三台版本一致：`mesh-llm 0.65.0-rc2+splitgguf.5`。
- 本地检查通过：`cargo fmt --all -- --check`、`cargo check -p mesh-llm`、`just build`。
- 两台 PGX 均完成：`just build cuda 121`。
- 运行日志目录：`~/.mesh-llm/run-logs/20260428-203349/`。
- Mac 状态：`3131` 可访问，`peers=2`，模型处于 `Loading`。
- `9337/v1/models` 在 `llama_ready=false` 时返回空列表，需等待 405B 模型加载完成。
- 两台 PGX 日志均出现正确改写：

```text
Rewrote REGISTER_PEER: peer_id=0 127.0.0.1:<host_port> -> 127.0.0.1:<local_tunnel_port>
Rewrote REGISTER_PEER: peer_id=1 127.0.0.1:<host_port> -> 127.0.0.1:<local_tunnel_port>
```

### 注意事项

- 这是 RPC tunnel / peer discovery 的通用修复，不是 Hermes 专用。
- Hermes 405B Q8 加载时间很长，加载期间前端可能持续转圈。
- 判断是否可用以 `llama_ready=true` 和 `/v1/models` 有模型为准。

- 不要把 IP、密码、token、SSH 私密信息写入 git 跟踪文件。
